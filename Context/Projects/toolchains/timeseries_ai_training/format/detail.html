ADAS Log-to-AI Workbench

A CANape/MF4-to-AI workbench designed to help ADAS module owners convert familiar vehicle logs into AI-ready datasets, training runs, and exportable model artifacts.

01

Context

The commercial-vehicle ADAS team collected real-vehicle and calibration data mainly through CANape in MF4 format. Module owners were comfortable with signal review and calibration workflows, but AI experimentation required a repeatable path from logs to datasets, model training, evaluation, and export.

This workbench was designed as a transition platform: keep the familiar measurement data flow, then add structured AI preparation and model-artifact export on top.

02

Problem

Without a common workflow, each engineer would have to repeatedly write separate scripts for trimming, normalization, signal selection, training, evaluation, and export. That makes AI learning slow, inconsistent, and hard to compare across modules.

03

Constraint

  • The workflow had to start from existing CANape/MF4 measurement data.
  • Module owners needed to plug in their own Python model scripts rather than adopting a full ML platform from day one.
  • The tool had to support learning and experimentation before formal team-wide operational adoption.
  • Public material could not include proprietary logs, signal names, datasets, model parameters, or validation criteria.
04

Decision

I designed a GUI-based workbench where engineers can load MF4 logs, select and preprocess signals, apply trimming/normalization, run module-specific Python training scripts, inspect results, and export model artifacts for downstream validation workflows.

05

Impact

Lower barrierMade AI experimentation approachable from the team’s existing vehicle-log workflow.
RepeatabilityStandardized the path from raw MF4 logs to datasets, training, and export artifacts.
Transition valueHelped module owners learn model behavior without needing a full ML platform immediately.
06

Implementation

  • MF4/CANape log loading workflow.
  • Signal selection, trimming, normalization, and dataset preparation pipeline.
  • Training-page structure for independently written module-owner Python scripts.
  • Result visualization and model evaluation page.
  • ONNX and MATLAB-oriented export path for later validation workflows.
CANape MF4 logs ↓ Preprocess: trim / normalize / select signals ↓ Train: module-owner Python model script ↓ Result review ↓ ONNX / MATLAB-oriented export
