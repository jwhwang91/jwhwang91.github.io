Turning a Legacy Calibration Setup into an ADAS AI Validation Loop

Python / C++ / CANape / VN1640 / XCP over CAN-FD / Shared Memory / ONNX

This project converted an existing CANape/VN1640 laptop-based calibration workflow into a minimum viable ADAS AI validation loop for real-vehicle experimentation.

The production ECU technically supported Ethernet communication, which would have been the preferred path for a high-bandwidth AI validation workflow. However, the available field development process was still based on CANape, VN1640, and a laptop used for flashing, calibration, and measurement.

Context

The limitation was not the ECU itself, but the available test equipment and team workflow. Vector Ethernet interface hardware was not available in the test environment, so the project repurposed the tools already present in field testing.

The engineering value was not in choosing XCP as the ideal transport. It was in converting a legacy calibration workflow into a practical AI validation loop under real production-development constraints.

Problem

The team wanted to explore AI-based ADAS validation on real vehicles, but the field test workflow was still built around laptop-based calibration and flashing.

Constraint

The mass-production ECU supported Ethernet, but the team did not have the Vector Ethernet interface hardware required to use that path. The available setup was CANape + VN1640 + laptop.

Decision

I chose not to wait for ideal infrastructure. I reused the existing calibration workflow and built a bypass bridge on top of XCP over CAN-FD.

Impact

Enabled real-vehicle AI model validation without new Ethernet interface hardware, connected legacy calibration practice with learned-model experimentation, and created a stepping stone toward future Ethernet/HPC-based validation.

Implementation

  • XCP-over-CAN-FD access from production ECU through the available calibration setup.
  • C++ communication backend for real-time vehicle-facing data movement.
  • Shared-memory IPC between backend and Python layer.
  • Python/PyQt visualization and control UI.
  • ONNX/Python model runtime.
  • Minimal public-safe data set to fit bandwidth and timing constraints.

AI-Assisted Development, Human-Owned Architecture

This project was developed using an AI-assisted coding workflow, but the engineering ownership remained human-driven.

I defined the initial requirements, designed the system architecture, decomposed the feature set, specified module boundaries, reviewed generated code, debugged runtime failures, tested the tool against real workflows, and iterated the implementation until the bypass loop became usable.

The AI coding agent was useful for accelerating implementation, refactoring, and UI/tooling iteration, but it frequently produced incomplete or brittle code. The practical value came from repeatedly converting vehicle-domain requirements into precise implementation tasks, executing the code, identifying failures, feeding back corrections, and making the final engineering decisions manually.

The result was a working ADAS AI validation bypass prototype delivered in approximately two months under legacy field-equipment constraints.
